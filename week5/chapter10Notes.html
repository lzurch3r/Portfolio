Errors, Exceptions, and Warnings

Three types of errors:

Programmer errors are a programmer�s responsibility to work out and minimize. User errors, while not directly a programmer�s fault, still are the programmer�s responsibility to avoid.

An exception is an error that produces a return value that can then be used by the program to deal with the error. An exception will also produce a stack trace � a sequence of functions or method calls that lead to the point where the error occurred. A stack trace will work backward from the point at which the error occurred to identify the function or method that started the sequence.

A warning can occur if there�s an error in the code that isn�t enough to cause the program to crash. May lead to problems if the program still runs, as it may do so incorrectly. Sometimes, a warning icon will display in the browser to indicate an error; other times, you may need to open the console to view an error.

When a runtime error occurs in the browser, the HTML will still appear, but the JS code will stop working in the background.

The Importance of Testing and Debugging

Ensure that when code fails, it fails loudly. Do not let a bug go unnoticed. But also don�t inhibit user experience. Code must be tested rigorously to give the best possible experience.

Strict Mode

Strict mode considers deprecated coding practices as �poor style� rather than actual errors. Writing code with strict mode turned on increases the chance of errors, since it will throw exceptions if it detects sloppy code practices. To use strict mode, add the following string to the first line of a JavaScript file:

'use strict';

 

You can also use strict mode inside functions on a per-function basis. Add the same string above to a function and it will apply to everything inside it.

Linting tools such as JS Lint, JS Hint, and ES Lint can be used to test JS code quality. They are designed to highlight any sloppy programming practices or syntax errors. They are useful for enforcing a programming style guide, and it ensures that everyone working in a team follows the same conventions. Linting can be implemented in software, as a plugin, or online.

Debugging in the Browser

The most basic form of debugging is the alert() function. It stops a program from running until OK is clicked. The use of this function is discouraged for debugging today.

console.log() is much more common now. console.trace() will log an interactive stack trace in the console. It will show the functions that were called in the lead up to an exception occurring while the code is running.

Most modern browsers also have a debugging tool that allows you to set breakpoints in your code. Debugging documentation for major browsers:

https://developer.mozilla.org/en-US/docs/Tools/Debugger

https://docs.microsoft.com/en-us/microsoft-edge/f12-devtools-guide/debugger

https://developer.chrome.com/devtools/docs/javascript-debugging

https://developer.apple.com/library/mac/documentation/AppleApplications/Conceptual/Safari_Developer_Guide/Debugger/Debugger.html

One of the most useful commands is the debugger keyword. This will create a breakpoint in the code that will pause the execution of the code and allow you to see where the program is currently up to. You can also hover over any variables to see what value they hold at that point.

NOTE: Remember to remove any references to the debugger command before shipping any code, otherwise the program will appear to freeze when people try to use it!

Exception Handling

It is not good to terminate the program every time an exception occurs, or it will look like the program crashes, reflecting badly on the programmer. Handle exceptions gracefully by catching the error.

If you suspect a piece of code will result in an exception, wrap it in a try block. It will run the code as normal, but if an exception occurs, it will pass the error object that is thrown onto a catch block. The error object is passed as a parameter to the catch block.

A finally block can be added after a catch block, and will always be executed after the try or catch block. Use it if you want code to run in any case.

Tests

Writing good tests helps identify errors early on. They can simply be functions that tests that a piece of code runs as it should.

Test-driven Development (TDD) is the process of writing tests before any actual code. While the tests will initially fail, your next step is to write code to make the tests pass. Then the code is refactored to make it faster and more readable, and remove any repetition. �This is how workflow should be:

1.     Write tests (that initially fail)

2.     Write code to pass the tests

3.     Refactor the code

4.     Test refactored code

5.     Write more tests for new features

This is often referred to as the �red-green-refactor� cycle of TDD. TDD is considered to be best practice. Still, any tests are better than no tests at all.

Jest

Jest is a TDD framework, created by Facebook, that makes it easy to create and run tests by providing helper methods for common test assertions.
